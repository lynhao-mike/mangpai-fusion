# Rule Family Discovery Audit

审计范围：

- theory/ziping/index.yaml
- theory/tiaohou_ditiansui/index.yaml

审计性质：只读语义审计。本文只识别同源、近义、重复表达与重复投票风险，不修改规则、权重或 Schema。

审计口径：不按文件结构或 topic 机械统计，而按规则表达的理论核心归并。同一规则可在实际引擎中只属于一个主 family，但在风险说明中可提示与其他 family 的交叉重叠。

总体发现：

- 子平规则库共 57 条，滴天髓/调候规则库共 255 条。
- 滴天髓/调候规则数量约为子平的 4.47 倍，且在“调候、顺逆流通、从化、清浊、中和偏枯”等理论核心上存在密集拆分。
- 多个理论核心已被拆成多个相邻规则或跨库重复表达，若未来投票层按 rule 粒度直接累加，可能出现同一理论被重复计票。

# 3. 最可能影响校准结果的Family

## 1. FAM-004 调候 / 寒暖 / 燥湿 / 季令气候

影响未来 30 案例校准的方式：

- 该 family 横跨子平穷通宝鉴十干取用与滴天髓寒暖燥湿规则，数量最大且同向性强。
- 若未来 30 个案例中出现季节寒暖明显的命局，调候规则会在多个入口同时命中，导致系统把“季令气候”误判为多条独立证据。
- 校准结果可能表现为：调候相关规则命中率虚高、权重被持续上调、扶抑/格局/六亲等规则被相对稀释。

建议：调候应设 family-level cap 或 evidence de-dup；同一寒暖燥湿证据链只提供一次主票，其余规则作为解释层或细分标签。

## 2. FAM-011 从格 / 从象 / 从势 / 从旺 / 从强 / 从财杀儿

影响未来 30 案例校准的方式：

- 子平对特殊格只有总纲式规则，滴天髓拆分为大量从象/从旺/从强/从气/从势/从财/从杀/从儿规则。
- 若未来样本中少量特殊格判断命中，滴天髓规则数量会制造“多票一致”的假象。
- 校准结果可能表现为：特殊格判断被过度相信，普通扶抑框架被过早排除，从格误判成本上升。

建议：特殊格应先进行 family-level gate；gate 未通过时，所有从格细分规则不得参与主投票，只保留为待核验候选。

## 3. FAM-001 用神 / 取用 / 喜忌 / 成败救应

影响未来 30 案例校准的方式：

- 用神取裁是子平核心，同时滴天髓通过“不拘名美名恶、取用跨类别、中和补偏、顺逆”等表达重复参与。
- 如果按 rule 粒度投票，用神判断可能既作为上游判断，又在格局、事业、财富、运岁中重复出现。
- 校准结果可能表现为：用神相关规则全面高置信，后续错误会呈系统性扩散，而非单点错误。

建议：用神类规则应拆分为“定位、成败、救应、应用域”四层，并在投票层只允许上游定位得一票。

## 4. FAM-005 顺逆 / 顺势 / 流通 / 通关 / 源流

影响未来 30 案例校准的方式：

- 顺逆、流通、通关、源流多为同一气机连续性理论，容易被不同表述多次命中。
- 对事业、财富、健康、运岁都有解释力，因此在多域报告中容易重复加权。
- 校准结果可能表现为：所有“气机顺”的案例被整体抬高，所有“阻隔”的案例被整体压低，细节差异被吞没。

建议：流通类规则应建立 shared evidence key，例如“气机连续性/阻隔点/通关物”，避免每条规则独立加权。

## 5. FAM-013 月令 / 时令 / 旺相休囚 / 四时进退

影响未来 30 案例校准的方式：

- 月令既参与用神定位、旺衰判断、调候判断，又参与格局高低。
- 如果月令证据在多个 family 中都作为独立票，系统会高估“月令一致性”的信息量。
- 校准结果可能表现为：月令强相关规则在所有案例中稳定命中，形成结构性权重偏置。

建议：月令应作为公共 evidence feature，被多个规则引用但不可重复计作独立理论票。

## 审计结论

本次审计发现，两个目标规则库并非简单并列补充，而是在多个理论核心上存在语义重叠。最高风险来自三类结构：

1. 上游基础理论被多个下游规则反复引用：月令、旺衰、用神、中和。
2. 同一理论在滴天髓内部被密集拆分：从格、化格、清浊、顺逆、调候。
3. 子平总纲与滴天髓细则数量不对称：子平一条总纲可能对应滴天髓十余条细分规则。

因此，在未来投票与校准中，应优先引入 family-level 去重、shared evidence key、family cap 与上游/下游分层投票机制。否则未来 30 案例校准可能把规则数量误识别为理论强度，导致高频 family 权重虚高，低频但关键的规则被稀释。
