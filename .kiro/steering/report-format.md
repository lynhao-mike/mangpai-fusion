# 案例报告与 analysis.md 格式规约（repo-local）

> 本文件由命主反馈固化（2026-05-25）。
> 原 v1.3 strict-schema 模板渲染稿（master/client 双版 · 含大量 HTML 注释 + 重复反馈位 + 嵌套表格）**已废弃**。
> 改回 **v1.0 自由 markdown 格式**——与 cases C-2026-001 ~ C-2026-014 同结构。

---

## 一、`reports/C-YYYY-NNN-{干支}-report.md` 的 10 段结构

```
一、八字盘速览           （排盘信息表 + 大运排布全 9 运 + 关键结构 5-8 条）
二、4 派各自定性（速览）  （4 派对照表 + 一致性评级）
三、共识层断语            （4 派全支持，★★★★+，含证伪条件）
四、互补层断语            （多派同向，含联立加权计算式）
五、独门层断语            （单派独有，含慎用提示）
六、应期总表              （已过 + 未来；带证伪条件、状态列）
七、命主画像版            ★ 见下方"§七 必含规约"
八、风险提示              （应期误差 + 决策提示 + 派别局限）
九、问命主的回测问题      （让命理师顺序提问）
十、归档信息              （case_id + 指纹 + 文件路径）
```

**禁用**：
- ❌ 大量 HTML `<!-- §X-START -->` / `<!-- §X-END -->` 边界注释
- ❌ 每条断语挂 `反馈：[S-...] [ ]` 反馈位（这是 v1.3 反馈采集机制，污染可读性）
- ❌ master / client 双版分裂（用单一 `-report.md` 即可，命理师按需口头筛选）
- ❌ "三层 gate / passed_layers / 道门" 等机器内部术语暴露给最终读者
- ❌ 每条做功路径的冗余 boilerplate（例：来源 + 证伪 + 反馈连续 3 行重复）

**必含**：
- ✅ 双轨置信度：每条断语必带 `★N (X%)` 双轨表达
- ✅ 证伪条件：每条互补/独门断语必有"若 XXX 则失验"
- ✅ 派别标签：每条断语开头必带 `[派别]` 或 `[共识·X 派]` 或 `[互补·X+Y]`
- ✅ 应期表：已过 / 未来分段，每条带证伪条件

## 二、§七 命主画像版（必含规约）

> §七 是 **"命主截止当前状况的总览"**——不是工程元数据，不是 [AI-polish] 锚点段。
> 它是**让命理师一眼就能向命主复述的画像板**。

### 必含五维分级

```
学业 · 事业 · 婚姻 · 财富 · 官运
```

> 五维次序固定。**不要替换为"子女 / 原生家庭 / 健康"**——后三者属于其他章节，
> 画像版只承担五维本身。健康/风险归 §八，子女/原生家庭归独立段或合并到分析过程。

### 财富分级硬约束

财富域**必须明确收入水平**，按命主层级选用计法：

| 命主层级（按 wealth_ceiling）| 收入计法 | 例 |
|---|---|---|
| 贫困级 / 小康下 / 小康中 / 小康上 / 工薪 / 普工 / 温饱 / 贫寒 / 极贫 | **月收入** | 1.5 万 / 月 |
| 中富下 / 中富中 / 中富上 / 大富下 / 大富中 / 大富上 / 巨富 | **年收入**（可附月收入参考） | 100-200 万 / 年（兑现期） |

**写法范例（中富下层）**：
```
【财富】L9 中富下（500-1000 万 / 千万级资产）★★★★ (85%)
   · 一生天花板：中富级·中（L9-L10 顶）
   · 资产量级：500-1000 万 / 千万级
   · 当前阶段（XX 大运）：
     · 月入参考（普通工薪计法）：1.5-3 万 / 月
     · 年入参考（富人计法）：100-200 万 / 年
   · 趋势：📈 上行 / ➡️ 平稳 / 📉 下行
```

**写法范例（小康层）**：
```
【财富】L7 小康中（30-50 万年入）★★★ (72%)
   · 一生天花板：小康中（L7-L8 顶）
   · 当前阶段：
     · 月入：3-5 万 / 月  ← 主报项
     · 年入：30-50 万 / 年（含奖金）  ← 辅助
   · 趋势：➡️ 平稳
```

## 三、`cases/C-YYYY-NNN/analysis.md` 格式

> analysis.md 是**命理师查阅档**，命主一般不读。
> 必须**保留 4 派各自详细推理 + 共识/互补/独门/冲突仲裁全过程**。

按 C-2026-001 / C-2026-002 ~ C-2026-014 同结构：

```
命盘速览 + 关键结构表
Step 1 · 4 派各自定性（每派一节，含规律 ID + 引用）
Step 2 · 共识层匹配（CONSENSUS）
Step 3 · 互补层联立加权（COMPLEMENTARY）（含加权计算式）
Step 4 · 独门层（EXCLUSIVE）
Step 5 · 冲突层（CONFLICTS）（含仲裁记录）
Step 6 · 应期总表 / 流年回溯
Step 7 · （当前流年评估，可选）
Step 8 · 仲裁日志
Step 9 · 风险与不确定性（含命主请重点回测的关键问题）
元数据
```

**禁用**：
- ❌ 把 analysis.md 写成"v1.3 流水线产物的索引页"（仅指向 findings/*.json + reports/*.md）
- ❌ 用结构化 JSON / 模板渲染输出替代人类可读的推理叙事

**必含**：
- ✅ 4 派各自的格局判定 + 关键签字 + 推理路径
- ✅ 联立加权计算式（max + Σ × 0.15）
- ✅ 仲裁日志（即使为空也要列）
- ✅ 信息缺失列表 + 派别仍存争议点 + 命主回测问题清单

## 四、与 v1.3 引擎的关系

- v1.3 strict-schema 引擎产出的 `findings/*.json` 与 `statement_index.json` **保留作为底层数据**，供 feedback_ingest / feedback_loop 自动重算
- 但**面向人的输出**（reports/*.md + cases/*/analysis.md）一律采用 v1.0 自由 markdown 格式
- 二者并存：机器层走 JSON 闭环，人类层走自由 markdown
