# analysis.md 长期输出规范候选模板

> 本文件基于以下三类既有产物抽取长期规范：
>
> - 历史 `reports/C-2026-019-乾-辛巳壬辰甲寅乙丑-report-v2.md`：曾作为命主可读、反馈校准、已知事实与白话画像样例；当前不再作为默认用户报告入口。
> - `cases/C-2026-001-乾-庚申戊寅壬子辛丑/analysis.md`：技术推理日志完整，四派分层、冲突仲裁、应期表齐备。
> - `cases/C-2026-018-坤-乙丑戊寅乙酉乙酉/analysis.md`：反馈校准、错误撤销、黑名单规则、长期流年回溯最完整。
>
> 推荐将 `analysis.md` 定位为“命理师内部技术归档 + 可回测证据链”，并与默认命理师报告互相追踪；除非明确命令，不生成命主/用户可读表达。

## 1. 结构差异提炼

| 来源 | 定位 | 优点 | 不宜直接照搬点 | 应纳入长期规范的要素 |
|---|---|---|---|---|
| 历史 `report-v2.md` | 命主可读报告旧样例 | 已知事实、修正摘要、婚姻专项、风险提示、回测问题、白话画像完整 | 已不作为当前默认产物；推理链不够展开，不适合作为内部分析全文 | known_facts、校正摘要、回测问题、报告关联路径 |
| `C001 analysis.md` | 内部推理日志 | Step 结构清晰：四派定性、共识、互补、独门、冲突、应期、仲裁、风险 | 对反馈校准与规则撤销记录较弱 | 固定 Step 主干、仲裁表、置信度与派别归属 |
| `C018 analysis.md` | 反馈校准型内部归档 | 有错误撤销、黑名单、长期流年回溯、当前流年专项 | 全量流年回溯很长，不适合每案默认展开 | 校准日志、禁用规则、当前年专项、缺失事实清单 |

## 2. 候选模板方案 A：技术推理完整型

### 适用场景

- 新案初断。
- known_facts 很少或完全为空。
- 需要保留完整推理链，供后续反馈摄入与规则校准。

### 骨架

```markdown

> 本文件为四派融合分析的内部技术归档，不是命主可读报告。

| 项 | 路径/内容 | 状态 |
|---|---|---|
| input | cases/{case_id}/input.md | 已校验 |
| report | reports/{case_id}-analyst-report.md | 已生成 |
| feedback | cases/{case_id}/feedback.md | 待回填 |
| statement_index | cases/{case_id}/statement_index.json | 已生成 |

## 1. 命盘速览

## 2. 关键结构抽取

## Step 1 · 四派各自定性

### 1.1 段派
### 1.2 杨派
### 1.3 高派
### 1.4 任派

## Step 2 · 共识层匹配（CONSENSUS）

## Step 3 · 互补层联立加权（COMPLEMENTARY）

## Step 4 · 独门层（EXCLUSIVE）

## Step 5 · 冲突层（CONFLICTS）

## Step 6 · 应期总表

## Step 7 · 仲裁日志

## Step 8 · 风险与不确定性

## 元数据
```

### 优点

- 与既有 `C001 analysis.md`、`C018 analysis.md` 兼容最高。
- 适合作为默认推理归档。

### 缺点

- 命主可读性弱。
- 如果不加 known_facts 与反馈字段，后续迭代校准成本较高。

## 3. 候选模板方案 B：反馈校准归档型

### 适用场景

- 已有命主反馈。
- 需要记录某条断语被升星、降星、撤销或列入黑名单。
- 需要对旧报告做 v2 / v3 修订。

### 骨架

```markdown

## 0. 案例状态与版本

| 字段 | 内容 |
|---|---|

| calibration_status | feedback-calibrated |
| known_facts_count | n |
| revoked_rules | ... |
| blacklisted_rules | ... |

## 2. 本轮修正摘要

| 修正项 | 原判断 | 反馈/证据 | 新判断 | 影响 |
|---|---|---|---|---|

## 3. 禁用/撤销规则记录

## 4. 四派重新定性

## 5. 共识/互补/独门/冲突重新计算

## 6. 应期回测表

## 7. 当前流年专项

## 8. 仍待验证问题

## 9. 元数据
```

### 优点

- 与 `C018 analysis.md` 的纠错/校准机制一致。
- 能清楚记录“为什么改”。

### 缺点

- 对初断案例略重。
- 如果没有反馈，会显得空。

## 4. 候选模板方案 C：报告-分析双轨映射型

### 适用场景

- 需要保证 `reports/` 与 `cases/` 双侧归档一致。
- 需要让每个报告结论都能追溯到分析断语与 statement_id。
- 适合自动化渲染与质量检查。

### 骨架

```markdown
# {case_id} · 分析过程（报告映射版）

## 0. 文件映射

| 报告章节 | analysis 证据段 | statement_id | feedback 字段 |
|---|---|---|---|

## 1. 报告结论索引

## 2. 断语证据链

| statement_id | 报告断语 | 派别来源 | 证据 | 置信度 | 证伪条件 |
|---|---|---|---|---|---|

## 3. 四派推理底稿

## 4. 应期映射表

## 5. 风险与边界

## 6. 输出一致性检查
```

### 优点

- 最适合长期工程化。
- 可以直接服务 `statement_index.json`、`feedback.md`、出口 linter。

### 缺点

- 单独阅读时不如 Step 型自然。
- 需要维护章节映射，人工书写成本略高。

## 5. 候选模板方案 D：推荐固定版 · 可回测技术归档型

### 推荐结论

建议长期固定采用方案 D。方案 D 是 A+B+C 的合并精简版：

- 保留 A 的 Step 推理主干。
- 引入 B 的 known_facts、校准日志、撤销规则。
- 引入 C 的 statement_id 与报告映射。
- 对每案默认输出中等长度；只有反馈案或专项案才展开 50 年流年回溯。

### 固定骨架


### 输出长度建议

| 案例类型 | 默认长度 | 是否展开长期流年 |
|---|---|---|
| 初断无反馈 | 250-450 行 | 只列关键 8-12 年 |
| 已反馈校准 | 450-800 行 | 展开命主反馈覆盖区间 |
| 婚姻/事业专项 | 300-600 行 | 只展开专项相关年份 |
| 争议复盘案 | 600 行以上 | 展开争议规则与撤销记录 |

## 6. 长期固定字段要求

### 6.1 必填字段

| 字段 | 目的 |
|---|---|
| case_id | 反馈摄入唯一键 |
| input_path | 可复核原始输入 |
| report_path | 对应命主可读报告 |
| feedback_path | 反馈回填入口 |
| statement_id | 每条可验证断语的唯一追踪号 |
| 派别来源 | 区分段/杨/高/任或融合结论 |
| 置信度 | 双轨模型：星级 + 百分比 |
| 证伪条件 | 避免铁口断语不可检验 |
| 缺失事实 | 明确置信度上限 |

### 6.2 断语最小单元

每条核心断语建议使用以下字段：

```yaml
statement_id: S-{case_seq}-{domain}-{number}
domain: 事业 / 财富 / 婚姻 / 健康 / 应期 / 学业 / 官命
claim: 可验证断语
school_sources: [段派, 杨派, 高派, 任派]
evidence: 原局/大运/流年/神煞/反馈
confidence_star: ★★★★
confidence_pct: 84
falsifier: 证伪条件
feedback_key: feedback.md 中对应问题
status: pending / verified / contradicted / revised
```

### 6.3 规则

1. `analysis.md` 不写纯鸡汤式结论；所有核心结论必须能追溯到派别、证据或 statement_id。
2. `reports/*.md` 可以白话化，但 `analysis.md` 必须保留证据链。
3. known_facts 为空时，最高置信度原则上不超过 ★★★★，除非是纯盘面强结构结论。
4. 神煞只能作为 boost 或旁证，不单独决定财富、婚姻、官命层级。
5. 冲突结论必须进入仲裁表，不允许隐性折中。
6. 被反馈证伪的规则必须写入撤销或黑名单段落。
7. 每个 case 的 `analysis.md`、`feedback.md`、`statement_index.json` 必须互相对齐。
