## 3. 当前结论

### 3.1 是否可以开始写 YAML？

**可以开始，但只能做“候选 YAML 草案 / 旁路 YAML”，不能直接并入生产 active 规则。**

理由：

1. 候选规则来源已经存在；
2. 权重矩阵已经同意；
3. 裁判契约与补强字段已经形成；
4. 案例回测说明方向可行；
5. 但 [`theory/SCHEMA.md`](theory/SCHEMA.md) 尚未扩展子平与滴天髓，因此第一步应先写 schema 扩展草案或旁路候选 YAML，而不是直接写入正式规则库。

建议 YAML 阶段的安全入口：

- 新建候选规则旁路目录；
- 所有规则状态固定为 candidate；
- 不接入引擎；
- 不参与正式报告；
- 只用于人工审查与 schema 验证。

### 3.2 是否可以开始改生产代码？

**暂不建议直接改生产代码。**

理由：

1. 裁判模型契约尚未转成最终字段 schema；
2. 没有对应测试用例；
3. 未设计与现有 [`engine/application/pipeline_runner.py`](engine/application/pipeline_runner.py) 的旁路接入方式；
4. 未明确报告渲染层如何展示少数派、时间分段与权重版本；
5. 直接改生产代码有破坏现有四派流程的风险。

## 4. 推荐下一步

建议下一步做 **YAML/schema 准备层**，仍不改生产代码：

1. 起草子平 / 滴天髓规则 YAML schema 扩展文档；
2. 设计候选规则 YAML 的最小字段；
3. 设计领域权重 YAML 的最小字段；
4. 设计裁判模型契约 YAML/JSON 示例；
5. 通过人工审查后，再决定是否创建旁路 YAML 文件。

## 5. 阻塞项

进入正式生产代码前仍有以下阻塞项：

1. 未扩展规则 schema；
2. 未定义旁路加载器；
3. 未定义裁判模型数据类；
4. 未定义最小测试集；
5. 未定义报告层展示策略；
6. 未定义回滚策略；
7. 未明确现有 D1-D4 与新三专家体系的并行过渡方式。

## 6. 明确判断

- **可以开始写 YAML 草案 / schema 草案。**
- **不建议现在改生产代码。**
- **如果要进入代码实现，至少应先完成 schema 扩展草案、旁路 YAML 示例、最小测试计划。**
