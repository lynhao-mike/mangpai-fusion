# 裁判模型契约草案（审查态）

> 目标：把“子平 / 滴天髓 / 盲派”的外部输出统一成可仲裁接口，供裁判层进行后验合成。
>
> 状态：审查态草案；不写入 YAML；不改生产代码；不替代任何流派内部推理。

## 1. 契约目标

裁判模型契约的目标不是统一各派思想，而是统一各派的**输出外壳**，便于：

1. 记录各派独立判断结果；
2. 比较同义、互补、冲突关系；
3. 在不破坏派系隔离的前提下做后验仲裁；
4. 通过反馈积累动态调整各派在不同功能域的权重。

## 2. 最小统一输出结构

每个流派建议输出同一组最小字段，字段名固定，但字段内部内容可保留派系差异。

| 字段 | 必填 | 说明 |
|---|---|---|
| expert_system | 是 | 流派标识，如 `blind`、`ziping`、`tiaohou_ditiansui` |
| expert_name | 是 | 流派显示名 |
| domain | 是 | 功能域，如财富、事业、婚姻、健康、性格、学业 |
| claim | 是 | 该派对该域的核心断语 |
| polarity | 是 | `positive` / `negative` / `mixed` |
| confidence_raw | 是 | 流派内部置信度 |
| evidence_refs | 是 | 对应样板条目或原文引用 |
| axis_refs | 是 | 命中的统一判断轴 |
| scope_limit | 是 | 适用边界 |
| falsifiable | 是 | 可证伪表达 |
| notes | 否 | 额外说明 |

这个结构是“最小共识壳”，不是流派内部规则格式。

## 3. 裁判层输入结构

裁判层接收的输入建议包括三层：

### 3.1 专家阅读列表

```text
ExpertReading[]
```

每个元素就是上面的最小统一输出结构。

### 3.2 域级上下文

裁判层还需要接收一些对该域裁决必要的上下文：

- 当前分析主题：财富 / 事业 / 婚姻 / 健康 / 性格 / 学业
- 当前问题类型：总评 / 细项 / 事件 / 风险 / 时机
- 用户指定的权重先验
- 历史反馈权重
- 样本量与置信门槛

### 3.3 冲突上下文

若出现派系意见相反，裁判层需要记录：

- 冲突域
- 冲突来源
- 反向结论
- 证据差异
- 是否进入少数派提示

## 4. 裁判层输出结构

裁判层输出建议分为三部分：

### 4.1 主结论

```text
DomainConsensus
```

包含：

- domain
- primary_judgement
- support_judgement
- conflict_summary
- winner_expert
- confidence_final
- uncertainty_note

### 4.2 冲突记录

```text
CrossSchoolConflict[]
```

包含：

- conflict_id
- domain
- involved_experts
- conflict_type
- arbitration_reason
- winner
- loser
- output_strategy

### 4.3 追踪信息

```text
ArbitrationTrace
```

包含：

- 参与专家列表
- 权重来源
- 评分维度明细
- 少数派是否保留
- 是否触发降级输出

## 5. 评分维度定义

裁判层不直接投票，而是对每个专家阅读做加权评分。建议评分维度如下：

| 维度 | 说明 | 评分方向 |
|---|---|---|
| prior_domain_weight | 该派在当前功能域的历史先验优势 | 越高越优 |
| axis_relevance | 命中判断轴与当前问题的相关性 | 越高越优 |
| evidence_quality | 原文是否明确、条件是否完整、边界是否清楚 | 越高越优 |
| structural_specificity | 是否给出可操作结构而非空泛描述 | 越高越优 |
| falsifiability | 是否写出了可证伪条件 | 越高越优 |
| feedback_weight | 历史反馈支持度 | 越高越优 |
| conflict_penalty | 与其他派冲突的惩罚项 | 越高越不利 |
| domain_fit | 当前问题是否落在该派优势区 | 越高越优 |

## 6. 裁判规则草案

### 6.1 同义合并

若两派命中同一轴且方向一致：

- 合并为同一个主结论；
- 证据链保留多源引用；
- 提升置信度，但不删除少数派痕迹。

### 6.2 互补拼接

若两派关注不同层面但并不矛盾：

- 以一个派的结论为主骨架；
- 另一个派作为修饰、补充、边界说明；
- 输出“主结论 + 补充解释”结构。

### 6.3 冲突仲裁

若两派结论相反：

1. 先判断是否属于优先级冲突；
2. 再判断是否属于适用边界冲突；
3. 再看历史反馈权重；
4. 最后决定主胜方、并列显示或降级输出。

## 7. 只读边界

为了保持派系隔离，裁判层必须满足：

- 只读各派最终输出，不读内部推理链；
- 不写回各派规则库；
- 不修改各派输入逻辑；
- 不把统一轴强制注入各派内部判断步骤。

## 8. 建议的人工审查标准

进入结构化规则库或代码实现前，必须满足：

1. 每个流派都能按最小统一结构输出；
2. 每条输出都可映射到至少一个统一判断轴；
3. 每条输出都能写出 scope_limit 与 falsifiable；
4. 裁判规则能区分同义、互补、冲突；
5. 任何冲突处理都能解释为什么主胜方获胜；
6. 不存在跨派中间态互读；
7. 不存在裁判结果回写各派内部规则库。

## 9. 下一步建议

若继续推进，下一步适合做：

1. 把这个契约草案细化成字段级模板；
2. 再把每个功能域的权重先验整理成可审查矩阵；
3. 之后再考虑是否进入 YAML 结构化阶段。
