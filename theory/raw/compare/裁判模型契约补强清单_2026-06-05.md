# 裁判模型契约补强清单（审查态）

> 来源：[`theory/raw/compare/裁判模型样例回测报告_2026-06-05.md`](theory/raw/compare/裁判模型样例回测报告_2026-06-05.md) 暴露的契约缺口。
>
> 状态：审查态草案；不写入 YAML；不改生产代码；不直接作为最终接口。

## 1. 补强目标

回测显示，现有 [`裁判模型契约草案`](theory/raw/compare/裁判模型契约草案_2026-06-05.md) 已能承接基础的三派输出与后验仲裁，但真实案例中还需要补强以下能力：

1. 时间分段能力；
2. 置信度拆分能力；
3. 反馈状态追踪能力；
4. 权重版本追踪能力；
5. 少数派观点保留能力。

这些补强字段只属于裁判层外壳，不进入各派内部推理。

## 2. 字段补强总表

| 补强方向 | 建议字段 | 作用 | 来源案例启发 |
|---|---|---|---|
| 时间分段 | time_scope / phase_scope | 表达前段、后段、大运、流年窗口 | C-2026-001 婚姻“前婚多变 + 晚遇贵妻” |
| 置信拆分 | confidence_raw / confidence_merged / confidence_feedback | 区分单派、合并、反馈后的置信 | C-2026-009 学业多派同向增强 |
| 反馈状态 | feedback_state | 区分 hit / miss / no_data / skip / pending | C-2026-025 事业财富 hit、婚姻健康 no_data |
| 权重版本 | weight_profile_id / weight_profile_version | 追踪本次裁判使用的权重矩阵版本 | C-2026-028 反馈更正与动态调权 |
| 少数派保留 | minority_views / preserve_minority | 保留未胜出但有价值的观点 | C-2026-001 杨高婚姻冲突 |
| 仲裁理由 | winner_reason / loser_reason | 解释主胜方为什么胜出 | 多案例冲突仲裁 |
| 证据粒度 | evidence_items | 区分原文、候选规则、案例反馈 | 所有案例 |
| 输出策略 | output_strategy | 主胜、并列、降级、分段合成 | C-2026-001、C-2026-015 |

## 3. 时间分段字段

### 3.1 建议字段

```text
time_scope:
  type: static | dayun | liunian | phase | event_window
  label: 前段婚姻 / 壬午运 / 2026丙午 / 2025-2027窗口
  start_year: optional
  end_year: optional
  description: optional
```

### 3.2 使用场景

- 婚姻：前段不顺，后段改善；
- 事业：某大运上升，某流年震荡；
- 健康：某几年风险集中；
- 财富：阶段性进财但不稳定。

### 3.3 审查要求

如果结论含有“早年 / 中年 / 晚年 / 当前运 / 某年”字样，必须填写 `time_scope`。

## 4. 置信度拆分字段

### 4.1 建议字段

```text
confidence:
  raw: 0.0-1.0
  merged: 0.0-1.0
  feedback_adjusted: 0.0-1.0
  star: 0-5
  percent: 0-100
  reason: string
```

### 4.2 字段解释

| 字段 | 含义 |
|---|---|
| raw | 单个流派独立输出时的原始置信度 |
| merged | 裁判层合并多派同向证据后的置信度 |
| feedback_adjusted | 结合案例反馈后的置信度 |
| star / percent | 面向报告输出的双轨置信度 |
| reason | 为什么升降置信 |

### 4.3 审查要求

- 多派同向时，`merged` 可以高于 `raw`；
- 反馈命中时，`feedback_adjusted` 可高于 `merged`；
- 低样本时，禁止 `feedback_adjusted` 大幅上调。

## 5. 反馈状态字段

### 5.1 建议枚举

```text
feedback_state: hit | miss | no_data | skip | pending | partial
```

### 5.2 字段解释

| 状态 | 含义 |
|---|---|
| hit | 已反馈命中 |
| miss | 已反馈失验 |
| no_data | 用户未提供相关事实 |
| skip | 未来年份或不可验证项，暂不计分 |
| pending | 已提出但尚未回收反馈 |
| partial | 部分命中、部分缺失 |

### 5.3 审查要求

权重修正只能使用 `hit` / `miss` / `partial`，不能把 `no_data` / `skip` / `pending` 当作命中或失验。

## 6. 权重版本追踪字段

### 6.1 建议字段

```text
weight_profile:
  profile_id: domain-prior-2026-06-05
  version: review-draft-v1
  source: theory/raw/compare/领域权重矩阵初始数值建议_2026-06-05.md
  domain_weights:
    blind: 0.30
    ziping: 0.45
    tiaohou_ditiansui: 0.25
```

### 6.2 使用场景

每次裁判输出都应能回溯当时使用的是哪版权重，避免未来调权后无法解释历史报告。

### 6.3 审查要求

- 所有裁判输出必须记录权重版本；
- 权重更新后，历史输出不应被静默重算；
- 若重算，必须产生新版本记录。

## 7. 少数派保留字段

### 7.1 建议字段

```text
minority_views:
  - expert_system: gaopai
    claim: 晚婚得贵或晚景配偶福厚
    reason_to_preserve: 与主结论不完全冲突，可作为后段风险/机会提示
    confidence: 0.47
    output_visibility: report_note | internal_only | suppressed
```

### 7.2 使用场景

- 主胜方判婚姻不顺，但少数派提示晚福；
- 主胜方判事业适合平台，少数派提示冲突风险；
- 主胜方判健康风险低，少数派提示特定年份高风险。

### 7.3 审查要求

少数派观点不得被无理由删除；若被压制，必须记录压制原因。

## 8. 仲裁理由字段

### 8.1 建议字段

```text
arbitration_reason:
  winner_reason: string
  loser_reason: string
  conflict_type: priority | scope | evidence | timing | feedback
  output_strategy: primary_only | primary_with_minority | parallel | downgraded | phased
```

### 8.2 审查要求

- 冲突仲裁必须写明冲突类型；
- 主胜方必须有可解释理由；
- 败方不能直接丢弃，除非证据不足或已反馈失验。

## 9. 证据粒度字段

### 9.1 建议字段

```text
evidence_items:
  - evidence_type: source_excerpt | candidate_rule | case_feedback | runtime_finding
    ref: ZP-SAMPLE-012
    summary: 财旺身弱，难任财帛
    strength: high | medium | low
```

### 9.2 审查要求

后续不能只写字符串数组，应区分：

- 原文依据；
- 候选规则；
- 案例反馈；
- 引擎运行发现。

## 12. 结论

本补强清单建议把裁判模型从“能合并三派输出”升级为“能追踪阶段、反馈、权重版本和少数派观点”。这一步是进入结构化接口或代码实现前的必要审查层。
