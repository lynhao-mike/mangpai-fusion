## 一、仲裁三步法

```
Step 1 · 冲突识别 → Step 2 · 冲突分类 → Step 3 · 仲裁裁决
```

### Step 1 · 冲突识别

主分析器在收集 4 派对同一对象的结论后，自动检测：
```python
if 多派结论同向（结论文字虽不同但语义同向）:
    → 互补层处理（详见 mapping/complementary.md + engine/confidence.yaml 联立加权）
elif 多派结论相反（语义相反或结论值相反）:
    → 进入 Step 2 冲突分类
elif 仅 1 派有结论:
    → 独门层处理（详见 mapping/exclusive.md）
elif 无派别有结论:
    → 不输出（避免幻觉）
```

### Step 2 · 冲突分类

按 mapping/conflicts.md 的 L1-L4 分级判定：

| 级别 | 描述 | 处理 |
|---|---|---|
| **L1 显性冲突** | 结论明确相反（如"必离"vs"不破婚"） | 进入 Step 3 仲裁 |
| **L2 路径差异** | 同向但路径不同 | 不算冲突 → 互补层处理 |
| **L3 适用边界差异** | 触发条件不同 | 取交集 + 显式说明边界 |
| **L4 表述差异** | 同一现象不同叫法 | 查派别术语对照表 → 互补层 |

### Step 3 · 仲裁裁决

L1 显性冲突按以下规则仲裁：

#### 规则 1：领域权重优先

查 [`engine/domain-weights.yaml`](domain-weights.yaml) 取得该领域的 lead 派：
```
婚姻 → 杨派 lead
应期 → 任派 lead
格局 → 段派 lead
神煞 → 高派 lead
财运 → 段派 lead
学业 → 杨派 lead
健康 → 高派 lead
调候 → 杨派 lead
六亲 → 杨派 lead
```

→ lead 派结论胜出。

#### 规则 2：高强度规律优先

如果 lead 派的支持规律 strength < 0.7，而非 lead 派 strength ≥ 0.85：
- **仍由 lead 派胜出**（领域权重优先于单条规律强度）
- 但报告中标注："本结论 lead 派支持较弱（strength=0.X），请重视另一方意见"

#### 规则 3：历史命中率优先

如果某条规律历史命中率 < 50% 且样本 ≥ 5 例：
- 该规律降权（× 0.5）
- 即使 lead 派持有，也可能输给非 lead 派
- 计算：`adjusted_score = static × 0.5 × domain_weight`

#### 规则 4：显式呈现冲突

无论谁胜出，报告中必须**完整列出双方观点**：
```
[杨派主导] 不破婚 ★★★ (62%)
[高派持反对] 婚姻多变 ★ (47%) - 权重低于本结论
   - 原因：婚姻领域杨派 lead，高派 audit
   - 高派支持规律：G-LF-022 阳包阴格局
```

#### 规则 5：4 派一致 vs 3 派一致

| 派别一致情况 | 仲裁 |
|---|---|
| 4 派全一致 | 共识层 → ★★★★★ (90%+) |
| 3 派一致 + 1 派沉默 | 互补层 → ★★★★ (~85%) |
| 3 派一致 + 1 派反对 | **互补层胜，但反对方必须显式登记** |
| 2 派 vs 2 派 | 按领域权重 → lead 派一方胜 |
| 1 派支持 + 3 派反对 | 反对方胜（除非该派为 lead） |

## 二、跨派联立加权公式（再次明示）

互补层多派同向时，置信度联立计算：

```
joint_score = max(各派 score) + Σ (其余派 score × 0.15)
封顶 100。
```

详见 `engine/confidence.yaml` §五。

## 三、冲突场景胜负公式

```
winner_score = winner.static × winner.domain_weight - loser.static × 0.3
loser_score  = loser.static × 0.6
```

**示例**：

```
案例：婚姻领域
  杨派 G-XX (lead, weight=1.0): static=85, 持"不破婚"
  高派 G-LF-022 (audit, weight=0.4): static=78, 持"婚姻多变"

winner (杨):
  winner_score = 85 × 1.0 - 78 × 0.3
              = 85 - 23.4
              = 61.6 → ★★★ (62%)

loser (高):
  loser_score = 78 × 0.6
              = 46.8 → ★ (47%)

输出：
  [杨派主导] 不破婚 ★★★ (62%)
  [高派持反对] 婚姻多变 ★ (47%)
```

## 五、仲裁日志记录

每次仲裁必须登记到 `META/arbitration-log.md`（待 M7 创建）：
```yaml
- date: 2026-MM-DD
  case_id: C-2026-NNN
  conflict_id: CF-MAR-04
  domain: hunyin
  positions:
    yang: 不破婚 (score=85)
    gao: 婚姻多变 (score=78)
  winner: yang
  output: ★★★ (62%) yang  +  ★ (47%) gao 反对
  feedback_pending: 命主反馈应验后再确认仲裁是否正确
```

仲裁错误（命主反馈显示反对方才对）→ 触发：
1. 调整该派在该领域的 domain_weight
2. 更新对应规律的 hit_count / miss_count
3. 增量更新 CF 规则

## 六、仲裁性能要求

每次仲裁必须在分析报告中体现：
- ✅ 冲突识别耗时 ≤ 1 秒（语义判断）
- ✅ 仲裁规则查表 ≤ 0.5 秒
- ✅ 联立加权计算 ≤ 0.5 秒
- ✅ 输出格式化 ≤ 1 秒
- ✅ 总仲裁层耗时 ≤ 3 秒（不影响整体分析速度）

---

## 七、不可仲裁的情形

以下情形不进入仲裁：
- 4 派完全无对应规律 → 不输出，避免幻觉
- 4 派结论完全一致 → 直接共识层输出
- 用户在策略选择器中要求"单派模式" → 仅用指定派别，不仲裁

---

## 八、仲裁规则的演化

CF 规则不是固定不变。每完成一个案例后：
1. 检查仲裁是否成功（命主反馈应验/失验）
2. 失败 ≥ 3 例 → 调整对应 CF 规则
3. 新发现的冲突模式 → 新增 CF 规则

详见 `META/calibration-methodology.md` 升降级流程。
