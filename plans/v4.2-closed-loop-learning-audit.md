# v4.2 Enhanced Prediction System 闭环学习系统审计报告

> 审计角色：Principal Systems Auditor  
> 审计目标：验证系统是否形成真实 Closed-loop Learning System，而不是验证文件或模块是否存在。  
> 审计方法：沿真实生产路径追踪 Data Flow / Control Flow / State Flow / Learning Flow。

---

## 0. 最终结论

**结论：当前 v4.2 Enhanced Prediction System 不是完整闭环学习系统。**

它更准确的系统分类是：

> **Prediction System with Partial File-based Learning Loop**  
> 即：具备预测输出、反馈记录、离线批量学习、流派权重持久化与后续预测读取能力，但学习闭环只在“流派权重”维度部分成立；事件语义与概率校准尚未形成真实生产闭环。

对用户指定闭环：

```text
Prediction → Feedback → Learning → Weight Update → Future Prediction
```

审计判定：**B. 部分闭环**。

原因：

- **Prediction → Future Prediction**：存在，生产预测由 [`run_pipeline()`](engine/application/pipeline_runner.py:32) 进入预测层。
- **Feedback → Learning**：存在，但主要通过离线工具与文件记录，不是生产服务内自动闭环。
- **Learning → Weight Update**：对流派权重存在，见 [`update_school_weights()`](engine/application/minimal_learning_loop.py:284)。
- **Weight Update → Future Prediction**：部分存在，权重写入 [`engine/expert-weights.yaml`](engine/expert-weights.yaml:1)，后续由 [`load_school_weights()`](engine/infrastructure/weight_repository.py:35) 读取。
- **语义配置学习闭环**：不成立，生产路径仍使用硬编码 [`_DOMAIN_MEANINGS`](engine/application/prediction_signals.py:88)。
- **概率校准学习闭环**：不成立，校准器被调用但默认参数等价于 identity，且生产反馈不会自动训练 [`ProbabilityCalibrator`](engine/application/probability_calibrator.py:66)。

## 1. System Reality Audit

### 1.1 [`time_window_estimator.py`](engine/application/time_window_estimator.py:1) 是否真正参与最终 [`PredictionOutput`](engine/domain/prediction.py:116)

**判定：是，真实参与生产输出。**

证据链：

1. 生产编排在 [`run_pipeline()`](engine/application/pipeline_runner.py:117) 进入 prediction step。
2. [`run_pipeline()`](engine/application/pipeline_runner.py:123) 调用 [`build_prediction_output()`](engine/application/prediction_layer.py:118)。
3. [`build_prediction_output()`](engine/application/prediction_layer.py:160) 调用 [`_estimate_time_window()`](engine/application/prediction_layer.py:51)。
4. [`_estimate_time_window()`](engine/application/prediction_layer.py:69) 调用 [`estimate_enhanced_time_window()`](engine/application/time_window_estimator.py:1)。
5. 返回结果写入 [`PredictionOutput.time_window`](engine/domain/prediction.py:122)。

审计结论：

- 这不是旁路工具；它在 production prediction path 上。
- 它的输出确实进入最终 [`PredictionOutput`](engine/domain/prediction.py:116)。
- 但当前只返回 enhanced result 的 primary window，增强解释、候选窗口、推理轨迹若存在，并未完整进入输出结构。

---

### 1.2 [`event-semantics.yaml`](theory/prediction_models/event-semantics.yaml:1) 是否替代硬编码 [`_DOMAIN_MEANINGS`](engine/application/prediction_signals.py:88)

**判定：否。外部语义配置未进入生产预测路径。**

证据链：

1. [`prediction_signals.py`](engine/application/prediction_signals.py:88) 仍定义硬编码 [`_DOMAIN_MEANINGS`](engine/application/prediction_signals.py:88)。
2. [`extract_mp_signal()`](engine/application/prediction_signals.py:99) 在没有 [`GateResult.event_type_hypotheses`](engine/yingqi/types.py:150) 时，仍执行 [`_DOMAIN_MEANINGS.get()`](engine/application/prediction_signals.py:121)。
3. 全仓搜索显示 [`event_semantics_loader.py`](engine/application/event_semantics_loader.py:1) 未被 [`prediction_signals.py`](engine/application/prediction_signals.py:1)、[`fusion_engine_v2.py`](engine/application/fusion_engine_v2.py:1)、[`pipeline_runner.py`](engine/application/pipeline_runner.py:1) 导入。
4. [`extract_symbolic_events_from_config()`](engine/application/event_semantics_loader.py:146) 只存在于 loader 文件内部，没有生产调用者。

审计结论：

- [`event-semantics.yaml`](theory/prediction_models/event-semantics.yaml:1) 当前是配置资产，不是生产事实源。
- [`event_semantics_loader.py`](engine/application/event_semantics_loader.py:1) 当前是未接入生产路径的能力模块。
- 事件语义仍由硬编码 [`_DOMAIN_MEANINGS`](engine/application/prediction_signals.py:88) 决定。
- 此处属于 **Dead Code / Dead Config Risk**。

---

### 1.3 [`probability_calibrator.py`](engine/application/probability_calibrator.py:1) 是否真实校准输出

**判定：生产路径会调用校准器，但当前默认表现接近 bypass / identity。**

证据链：

1. [`build_prediction_output()`](engine/application/prediction_layer.py:139) 会在 `use_calibration=True` 时取得校准器。
2. [`build_prediction_output()`](engine/application/prediction_layer.py:149) 对每个候选事件调用 [`calibrate()`](engine/application/probability_calibrator.py:72)。
3. [`_get_calibrator()`](engine/application/prediction_layer.py:43) 固定加载 [`engine/calibration_params.json`](engine/calibration_params.json:1)。
4. 当前 [`engine/`](engine:1) 目录下没有 [`calibration_params.json`](engine/calibration_params.json:1)。
5. [`ProbabilityCalibrator.load()`](engine/application/probability_calibrator.py:235) 在文件不存在时返回默认 [`ProbabilityCalibrator`](engine/application/probability_calibrator.py:66)。
6. 默认 [`CalibrationParams.temperature`](engine/application/probability_calibrator.py:37) 为 `1.0`，Temperature Scaling 在 `T=1.0` 时等价于不改变概率。
7. [`build_calibrator_from_feedback_logs()`](engine/application/probability_calibrator.py:262) 存在，但不在生产 pipeline 内自动调用。

审计结论：

- 从 Control Flow 看，校准器不是完全旁路，确实被调用。
- 从 State Flow 看，缺少训练后的参数文件，实际输出大概率不变。
- 从 Learning Flow 看，反馈不会自动更新校准参数。
- 因此它是 **wired but untrained**，不是成熟概率校准闭环。

---

### 1.4 [`batch_rl_backtest.py`](tools/batch_rl_backtest.py:1) 的训练结果是否影响未来预测

**判定：流派权重维度会影响未来预测；事件语义维度不会。**

证据链：

1. [`update_school_weights()`](engine/application/minimal_learning_loop.py:284) 会读取当前权重并更新。
2. [`update_school_weights()`](engine/application/minimal_learning_loop.py:329) 将更新后的权重写回 [`engine/expert-weights.yaml`](engine/expert-weights.yaml:1)。
3. [`save_expert_weights()`](engine/infrastructure/weight_repository.py:28) 是权重写入函数。
4. 未来预测调用 [`build_final_prediction()`](engine/application/fusion_engine_v2.py:60)。
5. [`build_final_prediction()`](engine/application/fusion_engine_v2.py:77) 在未传入 `runtime_school_weights` 时调用 [`load_school_weights()`](engine/infrastructure/weight_repository.py:35)。
6. [`load_school_weights()`](engine/infrastructure/weight_repository.py:38) 读取同一个 [`engine/expert-weights.yaml`](engine/expert-weights.yaml:1)。

因此流派权重闭环成立：

```text
feedback / backtest → update_school_weights → expert-weights.yaml → load_school_weights → build_final_prediction
```

但存在三个严重限制：

- [`pipeline_runner.py`](engine/application/pipeline_runner.py:121) 调用 [`build_final_prediction()`](engine/application/fusion_engine_v2.py:60) 时没有传入 `runtime_school_weights`，闭环依赖隐式文件 fallback，而不是显式状态注入。
- [`domain_hint`](engine/application/fusion_engine_v2.py:65) 默认是 `事业`，[`pipeline_runner.py`](engine/application/pipeline_runner.py:121) 也没有传入真实领域，因此很多预测可能只读取默认事业域权重。
- [`update_event_semantics_rl_weights()`](tools/batch_rl_backtest.py:345) 明确写着“暂时跳过自动写入”，只更新 metadata，不会把语义学习结果写成可被生产预测消费的语义权重。

审计结论：

- 不是伪学习系统，因为权重确实可回流影响未来预测。
- 但也不是完整学习系统，因为只有流派权重部分闭环，且闭环隐式、粗粒度、领域路由不足。

## 2. Closed-loop Audit

### 2.1 指定闭环逐段判定

| 链路段 | 判定 | 证据 | 缺口 |
|---|---:|---|---|
| Prediction | 成立 | [`run_pipeline()`](engine/application/pipeline_runner.py:32) 调用预测层 | 预测输出与学习样本未强绑定 |
| Feedback | 部分成立 | [`log_prediction_feedback()`](engine/application/minimal_learning_loop.py:335) 可记录反馈 | 生产服务未强制收集结构化反馈 |
| Learning | 部分成立 | [`update_school_weights()`](engine/application/minimal_learning_loop.py:284) 可更新权重 | 校准与语义未自动学习 |
| Weight Update | 成立 | [`save_expert_weights()`](engine/infrastructure/weight_repository.py:28) 写入权重 | 缺少版本化、实验隔离、回滚策略 |
| Future Prediction | 部分成立 | [`load_school_weights()`](engine/infrastructure/weight_repository.py:35) 被 [`build_final_prediction()`](engine/application/fusion_engine_v2.py:77) 调用 | `domain_hint` 默认化导致领域错配风险 |

### 2.2 总体闭环判定

**判定：B. 部分闭环。**

理由：

- 流派权重形成“文件型闭环”。
- 概率校准形成“调用型开环”：被调用，但无自动训练状态。
- 事件语义形成“配置型断环”：配置存在，但生产预测不消费。
- 反馈与预测输出之间缺少强 schema 关联，无法保证每一次预测都能被可学习地回收。

## 3. Convergence Audit

### 3.1 系统是否会越用越准

**判定：不会稳定地越用越准，只可能在部分领域、部分流派权重上缓慢漂移。**

原因：

- [`update_school_weights()`](engine/application/minimal_learning_loop.py:284) 只更新 school-level 权重，无法学习具体规则、事件类型、时间窗、概率校准曲线。
- [`build_final_prediction()`](engine/application/fusion_engine_v2.py:77) 默认读取一个 domain 的权重，但 [`pipeline_runner.py`](engine/application/pipeline_runner.py:121) 没有传入真实 domain。
- [`ProbabilityCalibrator.update_from_feedback()`](engine/application/probability_calibrator.py:137) 存在，但未接入反馈摄入主路径。
- [`event-semantics.yaml`](theory/prediction_models/event-semantics.yaml:1) 的学习更新不参与生产输出。

### 3.2 是否存在收敛机制

**弱收敛，仅在流派权重层存在。**

已有机制：

- 权重更新有增量限制，见 [`update_school_weights()`](engine/application/minimal_learning_loop.py:308)。
- 权重更新后会归一化，见 [`update_school_weights()`](engine/application/minimal_learning_loop.py:323)。

缺失机制：

- 没有 validation holdout 的自动通过门。
- 没有 per-domain / per-event / per-time-window 的指标面板。
- 没有 calibration error 监控，例如 Brier score、ECE、可靠性曲线。
- 没有防止历史反馈污染生产权重的版本闸门。

## 4. System Classification

| 分类 | 是否符合 | 审计说明 |
|---|---:|---|
| Feature Collection | 超过 | 系统不只是功能集合，预测链路已接入生产 pipeline |
| Prediction System | 符合 | 具备三流派信号、融合、概率分布、时间窗输出 |
| Learning System | 部分符合 | 流派权重具备反馈后持久化与回读，但学习范围有限 |
| Self-Evolving Decision System | 不符合 | 缺少自动评估、版本闸门、回滚、策略选择与全链路可验证收敛 |

最终分类：

> **处于 Prediction System → Learning System 的过渡阶段。**

不是简单 Feature Collection；也不是 Self-Evolving Decision System。

## 6. Top 5 Architecture Risks

### Risk 1：事件语义外部化未生效

[`event-semantics.yaml`](theory/prediction_models/event-semantics.yaml:1) 与 [`event_semantics_loader.py`](engine/application/event_semantics_loader.py:1) 未接入 [`extract_mp_signal()`](engine/application/prediction_signals.py:99)，导致生产预测仍依赖硬编码 [`_DOMAIN_MEANINGS`](engine/application/prediction_signals.py:88)。

影响：配置以为可调，实际输出不变。

### Risk 2：概率校准被调用但未训练

[`_get_calibrator()`](engine/application/prediction_layer.py:38) 加载缺失的 [`engine/calibration_params.json`](engine/calibration_params.json:1)，[`ProbabilityCalibrator.load()`](engine/application/probability_calibrator.py:235) 返回默认参数。

影响：系统表面有 calibration，实质可能是 identity transform。

### Risk 3：学习闭环依赖隐式文件 fallback

[`pipeline_runner.py`](engine/application/pipeline_runner.py:121) 没有传入 `runtime_school_weights`，真实闭环依赖 [`build_final_prediction()`](engine/application/fusion_engine_v2.py:77) 自动读取 [`engine/expert-weights.yaml`](engine/expert-weights.yaml:1)。

影响：链路可用但不可观察、不可测试、不可做 A/B 隔离。

### Risk 4：领域权重路由可能错配

[`build_final_prediction()`](engine/application/fusion_engine_v2.py:65) 的 `domain_hint` 默认是 `事业`，而 [`pipeline_runner.py`](engine/application/pipeline_runner.py:121) 没有传入真实预测领域。

影响：婚姻、财运、健康等事件可能使用事业域权重，造成学习反馈污染。

### Risk 5：学习状态缺少治理边界

[`save_expert_weights()`](engine/infrastructure/weight_repository.py:28) 直接覆盖 YAML，缺少版本号、训练批次、验证指标、回滚点与并发保护。

影响：一次坏反馈或批量回测错误可能污染后续所有预测。

## 7. Next-stage Recommendations

### P0：把事件语义配置接入生产路径

目标：让 [`event-semantics.yaml`](theory/prediction_models/event-semantics.yaml:1) 成为事件语义唯一事实源。

动作：

- 在 [`extract_mp_signal()`](engine/application/prediction_signals.py:99) 中替换硬编码 [`_DOMAIN_MEANINGS`](engine/application/prediction_signals.py:88)。
- 调用 [`get_domain_meanings()`](engine/application/event_semantics_loader.py:58) 或 [`extract_symbolic_events_from_config()`](engine/application/event_semantics_loader.py:146)。
- 保留硬编码仅作为 loader 失败 fallback。
- 增加测试：修改 [`event-semantics.yaml`](theory/prediction_models/event-semantics.yaml:1) 后，预测候选事件必须变化。

### P0：显式传递领域与权重上下文

目标：消除隐式 fallback，让学习状态可测试。

动作：

- 在 [`pipeline_runner.py`](engine/application/pipeline_runner.py:121) 推断或传入真实 `domain_hint`。
- 调用 [`load_school_weights()`](engine/infrastructure/weight_repository.py:35) 后显式传入 [`build_final_prediction()`](engine/application/fusion_engine_v2.py:60) 的 `runtime_school_weights`。
- 在 [`PredictionOutput`](engine/domain/prediction.py:116) 或内部 trace 中记录 weight version。

### P1：把概率校准纳入反馈训练闭环

目标：让 [`ProbabilityCalibrator`](engine/application/probability_calibrator.py:66) 从“被调用”变成“被训练”。

动作：

- 在反馈摄入后调用 [`update_from_feedback()`](engine/application/probability_calibrator.py:137)。
- 定期运行 [`build_calibrator_from_feedback_logs()`](engine/application/probability_calibrator.py:262) 生成 [`engine/calibration_params.json`](engine/calibration_params.json:1)。
- 用真实 prediction probability 替代 [`build_calibrator_from_feedback_logs()`](engine/application/probability_calibrator.py:302) 中的权重均值代理。

### P1：引入学习发布门

目标：避免坏反馈直接污染生产预测。

动作：

- 每次批量学习产出 candidate weights。
- 在 validation set 上计算命中率、Brier score、ECE、domain-level lift。
- 只有指标通过阈值才覆盖 [`engine/expert-weights.yaml`](engine/expert-weights.yaml:1)。
- 保存旧版本用于回滚。

### P2：建立完整学习样本契约

目标：让每条反馈都能追溯到预测时的模型状态。

每条样本至少包含：

- `prediction_id`
- `case_id`
- `domain`
- `event_candidate`
- `raw_probability`
- `calibrated_probability`
- `time_window`
- `school_weights_version`
- `semantic_config_version`
- `calibration_params_version`
- `actual_outcome`
- `verdict`

## 8. Final Judgment

v4.2 Enhanced Prediction System 已经具备“预测系统 + 局部学习回路”的雏形，但尚未形成完整 Closed-loop Learning System。

最关键判断：

- [`time_window_estimator.py`](engine/application/time_window_estimator.py:1)：**已进入生产预测路径。**
- [`event-semantics.yaml`](theory/prediction_models/event-semantics.yaml:1)：**未替代生产硬编码语义。**
- [`probability_calibrator.py`](engine/application/probability_calibrator.py:1)：**被调用，但当前无训练参数时等价于未校准。**
- [`batch_rl_backtest.py`](tools/batch_rl_backtest.py:1)：**能通过权重文件影响未来预测，但只在流派权重维度成立。**
- 完整闭环：**B. 部分闭环。**

系统下一阶段不应继续堆功能，而应把“学习状态”变成显式、版本化、可验证、可回滚的生产依赖。只有当事件语义、概率校准、流派权重、时间窗评估都能从反馈中更新，并通过指标门影响未来预测时，系统才可升级为真正的 Closed-loop Learning System。
