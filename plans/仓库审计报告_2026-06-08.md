## 1. 执行摘要

当前仓库已经从早期“四派权重平铺 + 人工解释”的资料库，演进为一个可运行的命理师内部八字分析与反馈校准系统。主干形态是：问真排盘输入 → 入口校验 → D1-D4 串行推理 → 报告渲染与归档 → 断语级反馈摄入 → 规则生命周期与置信度校准。

本次审计判断如下：

| 维度 | 审计结论 |
|---|---|
| 架构定位 | 清晰。仓库本质是“实战调用接口 + 自迭代反馈环”，不是单纯理论语料库。 |
| 主流水线 | 基本闭环。入口校验、pipeline、report、statement_index、feedback_ingest 均已形成可追踪链路。 |
| 推理模型 | D1-D4 串行盲派融合已稳定；v1.5 多专家功能域裁判已作为旁路接入。 |
| 规则库 | 四派 legacy 规则、子平/滴天髓 production rules、raw candidate rules 已形成分层，但 schema 文档存在代际漂移。 |
| Prompt/模板 | 输入模板完整；报告模板可用；反馈模板和报告尾注存在过期表述。 |
| 工具治理 | 核心入口清楚，但工具索引检查失败，说明 [`tools/README.md`](../tools/README.md:1) 与实际工具文件已漂移。 |
| 反馈闭环 | 结构化反馈入口已是当前主入口；动态权重校准仍处于 pending，低样本下不能下“某派最准”的确定性结论。 |
| 最大风险 | 不是核心代码不可用，而是文档/模板/工具索引的事实源漂移会误导新 agent 或命理师。 |

1. 修复工具索引漂移：补齐问真 staging/promotion 系列工具在 [`tools/README.md`](../tools/README.md:1) 的分类。
2. 修复模板漂移：更新 [`templates/feedback.md`](../templates/feedback.md:1) 的 deprecated 入口，修正 [`templates/report-v1.3.md`](../templates/report-v1.3.md:200) 中“暂不启用自动裁判模型”的过期描述。
3. 补齐 schema 文档：把子平/滴天髓 production rule schema 与 parallel adjudication schema 纳入契约或 schema 文档，避免只存在于代码中。

## 3. 总体架构

### 3.1 系统定位

本仓库面向命理师实战调用，核心目标不是“生成漂亮命理文本”，而是：

1. 把八字输入转成结构化命盘事实。
2. 用四派盲派融合模型输出可追溯的断语。
3. 给每条断语挂接规则、证据、置信度、statement_id。
4. 通过反馈不断校准规则命中率、生命周期与置信度。
5. 在 v1.5 方向上，让盲派专家组、子平格局派、滴天髓调候派分别分析同一功能域，再由裁判模型综合。

### 3.2 主数据流

当前主数据流可概括为：

```text
问真八字 APP 排盘
  → templates/input-from-wenzhen.md
  → cases/C-YYYY-NNN-{乾/坤}-{干支}/input.md
  → tools/preflight.py
  → engine/pipeline.py / engine/application/pipeline_runner.py
  → tools/render_report.py
  → reports/*.md
  → cases/*/feedback.md
  → tools/feedback_ingest.py / tools/feedback_loop.py
  → theory/*/index.yaml + META/*
```

对应事实源：[`README.md`](../README.md:24)、[`AGENTS.md`](../AGENTS.md:46)、[`engine/contracts/00-OVERVIEW.md`](../engine/contracts/00-OVERVIEW.md:121)。

### 3.3 分层结构

| 层级 | 代表文件 | 作用 |
|---|---|---|
| 输入校验层 | [`tools/preflight.py`](../tools/preflight.py:1) | 解析与校验 input.md，输出 ParsedInput。 |

| 领域模型层 | [`engine/domain/analysis.py`](../engine/domain/analysis.py:181)、[`engine/domain/parallel.py`](../engine/domain/parallel.py:1) | 定义 AnalysisOutput、ExpertReading、AdjudicationResult 等可序列化模型。 |
| 规则加载层 | [`engine/application/production_rule_loader.py`](../engine/application/production_rule_loader.py:1)、[`engine/application/candidate_rule_loader.py`](../engine/application/candidate_rule_loader.py:1) | 区分 production rules 与 bypass candidate rules。 |

## 4. 推理流程审计

### 4.1 D1-D4 串行盲派融合

当前主流水线由 [`engine/application/pipeline_runner.py`](../engine/application/pipeline_runner.py:25) 编排，顺序为：

| 阶段 | 派别/维度 | 作用 | 下游依赖 |
|---|---|---|---|
| D1 | 段派 / energy | 做功、能量、体用、势党、贼神捕神 | D2、D3、D4、integrate |
| D2 | 杨派 / picture | 画面、财富、官命、婚姻、行业路径 | D3、D4、integrate |
| D3 | 任派 / yingqi | 三层门、触发器、应期候选 | D4、integrate |
| D4 | 高派 / pangzheng | 神煞、健康、灾厄、旁证补强 | integrate |
| integrate | 综合整合 | 生成 FinalConclusion 与 AnalysisOutput | report/render/feedback |

审计判断：D1-D4 是纵向约束架构，不是四派横向投票。D1 对 D2/D3 有边界约束，D3 受 D1+D2 限制，D4 作横向旁证。

### 4.2 应用层编排

[`engine/application/pipeline_runner.py`](../engine/application/pipeline_runner.py:25) 的核心顺序是：

1. `evaluate_energy(parsed)` 生成 energy findings。
2. `match_picture(energy, parsed)` 生成 picture findings。
3. `_extract_candidates(parsed)` 抽取应期候选并调用 `gate_yingqi(...)`。
4. `support_with_shensha(...)` 做旁证。
5. `integrate(...)` 合并 D1-D4。
6. 软附加 dual-engine/fusion findings。
7. 软附加 parallel_analysis。

审计判断：主流水线容错策略较稳健；production rules 和 parallel_analysis 异常通常不阻断既有四派 pipeline。但这也意味着旁路失败可能被弱提示跳过，后续应确保报告或 timing 中有足够可见性。

### 4.3 整合层

[`engine/application/integration.py`](../engine/application/integration.py:172) 将 energy、picture、yingqi、pangzheng 汇总为 [`AnalysisOutput`](../engine/domain/analysis.py:181)。同时，整合层会尝试加载子平/滴天髓 production rules，并把触发规则注入最终结论。

审计判断：production rules 已不是纯文档，而是进入最终断语体系；因此 schema 与规则变更必须纳入测试和报告追踪。

当前实现已落地以下核心部件：

| analyzer 协议/注册表 | [`engine/application/domain_analyzers.py`](../engine/application/domain_analyzers.py:1) | 已 landed。 |
| 旁路 runner | [`engine/application/parallel_domain_runner.py`](../engine/application/parallel_domain_runner.py:25) | 已 landed，支持 abstain fallback。 |

### 5.2 流派隔离

当前隔离边界设计正确：

- 盲派专家组内部可以使用 D1-D4 findings。
- 子平格局派和滴天髓调候派 production rules 由各自 YAML 进入。
- 跨专家体系只通过 [`ExpertReading`](../engine/domain/parallel.py:150) 进入裁判层。
- 裁判层只读 reading、confidence、evidence、weight profile，不读取各派私有中间态。

审计判断：这符合用户要求的“派系隔离”。后续风险在于：如果新增具体 analyzer 时直接读取其他专家 reading，将破坏隔离，应在 analyzer 接口和测试中继续加 guard。

### 5.3 裁判权重

[`engine/application/adjudication.py`](../engine/application/adjudication.py:50) 的默认权重已不再硬编码，而是读取 [`theory/raw/yaml/domain_weight_profile_2026-06-05.yaml`](../theory/raw/yaml/domain_weight_profile_2026-06-05.yaml:1)。该文件状态为 review_draft，不是 active production weight。

审计判断：这是正确的安全策略。当前权重可用于旁路裁判与报告展示，但不能据此形成“哪派最准”的最终结论；动态反馈权重仍 pending。

## 8. 反馈闭环与置信度模型

### 8.1 结构化反馈入口

当前推荐反馈入口是 [`tools/feedback_ingest.py`](../tools/feedback_ingest.py:1)。其行为包括：

1. 读取 case 的 feedback.md。
2. 读取 statement_index.json。
3. 解析 `[y] / [n] / [?] / [skip]`。
4. 通过 [`fanout_to_rules`](../tools/feedback_ingest.py:228) 把断语级反馈映射到规则级 verdict。
5. 调用 [`tools/feedback_loop.py`](../tools/feedback_loop.py:1) 的生命周期核心。
6. 写入 iteration log、snapshot、iteration-state。

审计判断：断语级 feedback → rule-level fanout 是当前闭环核心，设计方向正确。

### 8.2 置信度模型

[`engine/contracts/06-confidence-model.md`](../engine/contracts/06-confidence-model.md:1) 定义双轨置信度：

- 星级为命理师快速阅读层。
- 百分比为内部排序与裁判计算层。
- Beta posterior 切换需要足够反馈样本。
- 应期三层门会限制最高星级。

审计判断：模型兼顾理论先验和实战反馈；但当前动态权重与 Beta 切换仍受样本量限制，不能过早比较“哪个流派更准”。

## 12. 对“哪个流派更准确”的审计结论

当前仓库不能、也不应在低样本下给出“某一流派整体最准”的确定性结论，原因如下：

1. 反馈样本量仍不足，当前 N_eff 运行快照为 13，低于 Beta 切换和稳健比较所需规模。
2. 四派 legacy 体系与 v1.5 三专家体系不是同一比较颗粒：段/杨/任/高在主流水线中是纵向分工，盲派/子平/滴天髓在 v1.5 中是横向功能域专家。
3. “准确度”必须按功能域拆分：婚姻、财运、事业、健康、学业、性格可能由不同专家体系更擅长。
4. 反馈必须能归因到 expert_system、domain、rule_ids、reading/adjudication，否则只能校准规则，不能稳健校准专家体系。
5. 当前 dynamic_feedback_weighting 仍 pending，裁判权重主要是 review-draft 先验，不是反馈后验。

可行的下一步不是宣布胜负，而是继续推进：

- 把每条多专家 reading 和 adjudication 写入 statement_index。
- feedback_ingest 支持 reading/adjudication 级 fanout。
- 按 expert_system × domain × rule 聚合命中率。
- 在每个功能域达到足够 N_eff 后，用 Wilson/Beta 下界比较，而非裸命中率排名。

## 14. 审计结论

当前仓库的核心工程方向是正确的：主流水线已能从结构化输入生成内部命理师报告，并通过 statement_index 与 feedback_ingest 进入规则级校准；v1.5 多专家功能域裁判也已经以旁路方式接入，且保持了流派隔离和 review-draft 权重边界。

当前最需要处理的不是大规模重构，而是事实源收敛：工具索引、反馈模板、报告模板、schema 文档必须与已经落地的代码保持一致。只要先清理这些漂移点，后续再推进 reading/adjudication 级反馈归因，仓库就可以更稳健地回答“不同流派在不同功能域的实战表现”这一核心问题。
