7. 当前系统成熟度可评为：**工程骨架成熟，资料规模初步可用，验证闭环半成熟，统计判别尚不成熟**。

- 问真 98 条样本多处于 `cases/raw_feedback/parsed/` 与 staging/preflight/plan 层，尚未全部转为正式 `cases/C-.../` 案例。
- 首批 5 条也只是 dry-run 转案方案，明确写明“不创建任何正式目录”。
- 当前反馈多为既有事实/事件，不等同于对报告断语逐条 `[S-...]` 的命中/未命中标注。
- 可以比较“某条规则/某类功能域/某种事件类型”的表现，但还不能严谨得出“哪个流派整体最准确”。

## 3. 用于多流派断语验证的可行性评估

### 3.1 可以验证什么

当前 98 条问真样本可以支持以下验证：

1. 规则触发覆盖率
   - 某流派生产规则在真实排盘中的触发频率。
   - 哪些规则长期不触发或触发过宽。

2. 功能域覆盖率
   - 学业、事业、财富、婚姻、健康、性格等域是否都有可输出断语。
   - 子平与滴天髓是否在某些域长期缺席或重复输出。

3. 事件匹配验证
   - 对已知事实中的升学、婚恋、职业、健康、财务、家庭事件进行事后匹配。
   - 判断规则断语是否能映射到事实类型。

4. 多流派冲突登记
   - 当盲派、子平、滴天髓对同一功能域给出不同倾向时，可以记录冲突并人工裁决。

5. 初步命中率回测
   - 对每个正式 case 生成报告后，围绕 `statement_index.json` 中的 `[S-...]` 断语做 `[y]/[n]/[?]/[skip]` 标注。

### 3.2 暂时不能严谨验证什么

当前样本暂时不能直接支持以下结论：

1. 不能直接判定“哪个流派整体最准确”。
   - 因为 98 条中大部分尚未形成正式 case 与逐条断语反馈。
   - 样本来源、事件类型、年龄段、性别、质量等级可能存在偏差。

2. 不能直接做完整前瞻预测评分。
   - 当前反馈多是历史事实，适合回测，不等于未来预测验证。

3. 不能直接比较不同流派的原始理论优劣。
   - 当前比较的是“已抽象为规则并被工程实现的理论片段”。
   - 理论未被抽象或未被触发的部分不会出现在评分中。

4. 不能直接将 B 级样本全部视为高可信统计样本。
   - 10 条存在干支字符问题。
   - 部分样本仍需人工复核排盘、事件年份与事实字段。

### 3.3 可行性等级

| 验证目标 | 当前可行性 | 说明 |
|---|---|---|
| 规则触发覆盖率 | 高 | 只需正式 input 与 pipeline 输出。 |
| 功能域覆盖率 | 高 | v1.5 已有并行功能域结构。 |
| 回测事实匹配 | 中 | 需要把 raw_feedback 转正式 case，并整理反馈事实。 |

结论：

- 这 98 条样本对于“构建验证闭环”非常有价值。

## 5. 最终系统建设目标整合

最终目标不是简单让多个流派“混合输出”，而是构建一个可审计、可回测、可校准的多专家命理分析系统。

### 5.1 目标架构

最终系统应形成如下流程：

```text
问真排盘 + 已知事实
  → 正式 case/input.md
  → D1-D4 盲派主流程
  → 子平格局派独立分析
  → 滴天髓调候派独立分析
  → 功能域并行 reading
  → 裁判模型整合
  → 命理师报告 + statement_index
  → 逐条反馈标注
  → 规则命中率 / 流派权重 / 领域权重校准
  → 下一轮生产规则与裁判权重更新
```

### 5.2 核心原则

1. 派系隔离
   - 子平不能偷用盲派做功逻辑。
   - 滴天髓不能偷用子平格局成败逻辑。
   - 盲派综合组也不能把子平/滴天髓生产规则混入内部推理。
   - 只能在统一 `ExpertReading` 层汇合。

2. 功能域并行
   - 每个流派都要对同一功能域独立输出判断、证据、置信度、可证伪断语。
   - 裁判层按领域权重、证据质量、历史命中率进行整合。

3. 反馈驱动
   - 不能凭理论声望确定权重。
   - 必须通过真实案例反馈逐条校准。
   - 统计口径应从“规则命中率”扩展到“流派 × 功能域 × 事件类型”。

4. 保留少数派
   - 裁判输出主结论时，应保留少数派观点。
   - 尤其当少数派命中关键事件时，应支持后续权重提升。

5. 双轨置信度
   - 继续使用星级 + 百分比。
   - 百分比应逐步由先验/人工评估过渡到反馈后验。

### 5.3 分阶段目标

#### 阶段 A：验证集建设

目标：把 98 条问真样本转化为可运行验证集。

最小动作：

1. 先批准首批 5 条 dry-run 方案。
2. 创建正式 `cases/C-.../` 目录。
3. 每案生成 `input.md`、`analysis.md`、`feedback.md`、`statement_index.json`。
4. 运行 `python -m tools.preflight cases/.../input.md`。
5. 生成命理师报告。

#### 阶段 B：多流派回测

目标：让三大专家体系对同一功能域输出可比结果。

最小动作：

1. 对首批 5 条运行 pipeline。
2. 保留每个功能域下的 blind/ziping/tiaohou_ditiansui reading。
3. 在 `statement_index.json` 中记录每条断语的流派、领域、规则 ID。
4. 人工按真实反馈标注 `[y]/[n]/[?]/[skip]`。

#### 阶段 C：统计与权重校准

目标：从“能分析”进入“能自我校准”。

最小动作：

1. 按规则 ID 聚合 hit/miss。
2. 按流派 × 功能域聚合命中率。
3. 计算低样本保护下的 Beta posterior。
4. 只在样本量达到阈值后调整领域权重。
5. 对冲突案例写入 arbitration log。

#### 阶段 D：生产化收敛

目标：让系统可以持续处理新案例，并稳定输出可追踪报告。

最小动作：

1. 批量转案 26 条 READY staging。
2. 修复 4 条 blocked Top30。
3. 扩展至 98 条完整样本。
4. 更新源文件提取台账，反映 57/255 active 规则现状。
5. 修复工具索引与模板漂移问题。

### 6.1 主要风险

| 风险 | 影响 | 建议 |

## 7. 最终判断

当前项目已经从“单一盲派融合分析器”进化到“多流派专家并行 + 裁判整合”的早期生产形态。

最重要的进展是：

- 子平与滴天髓不再只是文档样板，已经进入生产规则层。
- 问真 98 条真实样本为回测体系提供了足够的启动样本。
- v1.5 的多专家功能域模型已经具备承接这些规则与样本的工程通道。

但当前还不应宣称系统已经能客观判断“哪个流派最准确”。更准确的成熟度表述是：

> 系统已经具备多流派自动化分析与回测校准的工程基础；当前最关键任务是把问真样本正式 case 化，并将历史反馈转化为断语级命中/未命中标注。完成首批 5 条与 26 条 READY 样本后，才能开始形成有限但有意义的流派 × 功能域准确性评估；完成 98 条并建立持续反馈后，才适合讨论稳定的流派权重与实战准确度排序。

因此，最终系统建设目标应聚焦为：

```text
不是让系统一次性判定哪个流派最好，
而是让每个流派在相同功能域、相同案例、相同反馈口径下持续竞争，
由真实反馈逐步决定规则权重、领域权重与裁判策略。
```
